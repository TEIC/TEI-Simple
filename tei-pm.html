2. Documenting a Processing Model

Where desired, the recording of the original rendition of source documents is catered for through use of the global attributes (rend, rendition and style) as well as the <rendition> element (see [[undefined COHQW]]). What is needed in some other instances is a method through which to document one or more possible 'processing models' for the document instance being encoded. In this case a processing model is a template description for how the element should be transformed in one or more particular outputs. It might say that this element should be displayed in italics or presented alongside its alternative (e.g. in the example of <sic> and <corr>), with no details on how or in what processing language this might be accomplished. It should be noted that any processing model documentation should be considered illustrative rather than restrictive. This is a method for documenting one or more possible processing models in abstract terms and it should be clear that processing of such documents is not limited to only these possibilities. This enables the creator of a TEI customization to specify how they intend particular elements could be processed.

This method of working assumes a workflow with at least three distinct parts and roles, in creating digital outputs from TEI texts. These roles may be held all by a single individual or multiple large teams depending on the scope and available resources. The minimal divisions that underlie this approach to documenting processing models are:

  1. an editor manages the text integrity and makes the high-level output decisions. These fall into (at least) two parts:
    1. Structural descriptions, e.g. 'should the original or corrected version be displayed by default', or 'is this a block level or inline component'
    2. Indications of appearance ('titles are in italics').
  2. a programmer takes the editor's specification, and the TEI text(s), and creates the input for a designer to make the output.
  3. a designer creates the output envelope (for example, and book layout using typesetting software, or a web site using their locally-supported web framework), making decisions in conjunction with the editor ('use Garamond font throughout' or 'every page must show the departmental logo')

It is crucial to understand here that neither the designer nor the programmer necessarily need have any understanding of the semantics of TEI elements or the structure of a TEI document. The programmer, particularly, can use the processing model rules to extract material from the TEI text in a relatively simple way. The method works for a variety of output formats, as the programmer simply has to provide implementations for a small number of model behaviours, and understand the well-documented features of CSS. The concepts addressed by the behaviours (e.g. block and inline) are generally in line with XSL-FO, and HTML; the catalogue of functions is given under the element <model>.

The following elements are used to document processing models:

A processing model may by documented by one or more <model> elements. If these are provided within a <modelGrp> element, or appear inside the <elementSpec> itself, then they document alternative processing models. The <modelGrp> is used to group elements all of which are alternatives for the same output. If multiple <model> elements are provided within a <modelSequence> element, then they document a single model composed of a sequence of models to be processed as a unit.

2.1. Processing Models

The <model> element is the basis of any processing model documentation. The <model> element is used to document the intended processing model for a particular element in an abstract manner which separates the documentation of intended processing from its implementation in whatever processing language is chosen. A required behaviour attribute which documents the intended processing behaviour and controls which processing function should be used to present this element in output. By default the behaviour specified by a <model> element is applicable to every occurrence of the element documented by the parent <elementSpec> within which it appears. However, a <model> element's scope may be modified using the predicate attribute to supply an XPath predicate determining the context under which this model is applicable.

The processing model notation allows each element to be assigned to one or more general categories (<model>), and optionally given an output rendition description (<outputRendition>). This allows a processor to know whether to handle the element or not, and broadly speaking how to display or otherwise process it. The <model> and <outputRendition> documentation are recorded in the TEI ODD customization. The editor or customization creator can document the intended output without having any knowledge about the implementation language or methodologies employed. The scope of a <model> and its behaviour may be limited by the predicate attribute. For a given <elementSpec>, there can be as many <model> statements as required. Multiple <model> statements are regarded as an alternation -- only one of them should be applied. This is comparable to a 'case statement' in some programming languages.

If the intention is for several <model> statements to be active at the same time, they can be enclosed inside a <modelSequence>. If it is desired to group sets of <model> statements without implying a sequence, for example grouping all of those for the same kind of output, then a <modelGrp> element may be used.

2.2. Output Rendition

Within each <model> there can be set of <outputRendition> elements. For example, with the simple case of handling of the <hi> element a customization writer may decide that this should be treated as an inline element and normally output in italics:

<model behaviour="inline">  <outputRendition>font-style: italic;</outputRendition> </model>

The behaviour attribute specifies the name of a model behaviour inline. This behaviour and its parameters can be documented in a using the <paramSpec> element. While these express different categories of output possibilities they do not specify the precise details of implementation since that is dependent on the processing language chosen. The <outputRendition> element uses CSS syntax to specify the appearance. It would have been possible here to simply add 'display: inline;' to the rendition, to indicate that it is inline, but the intention is to separate the structural decision from the presentation of its appearance.

It is common for <hi> to use a rendition attribute to indicate appearance. By default, this is ignored, but useSourceRendition can be used to indicate that it should be taken account of:

<model behaviour="inline"  useSourceRendition="true">  <outputRendition>font-style: italic;</outputRendition> </model>

In this case the <outputRendition> should be considered only a fallback and will be overridden by the rendition attribute if it exists. This uses the same behaviour as CSS; i.e. if rendition involves a change to a different property from 'font-style', then this local modification remains active.

2.3. Model Contexts

If the processing model is context-dependent, the predicate attribute can provide an XPath expression to document when different models apply. In document processing for the <q> element, it may be desired to indicate that when it is inside a paragraph it should be treated as inline (with an italic font style), but when it isn't inside a paragraph, it should be treated as a block.

<model predicate="ancestor::p"  behaviour="inline">  <outputRendition>font-style: italic;</outputRendition> </model> <model behaviour="block">  <outputRendition>left-margin: 2em;</outputRendition> </model>

The <model> elements here are alternatives and are so by default. It is only if the test (the predicate attribute) does not match the context will the second <model> be applied. The <model> elements could be grouped in a <modelGrp> element if there was a reason to do so, for example they are all for one particular form of output. This enables a convenient distinction between sets of models whose difference is not their predicate but their form of output.

2.4. Parameters for Model Behaviour

In the examples above TEI elements have been mapped to to simple block and inline concepts. There are other behaviours possible and these can have parameters which themselves need to be documented.

A more complex example is provided by processing the <choice> element; this is commonly used to provide a pair of alternates (e.g. <sic> and <corr>). An editor or customization writer may wish to document that the corrected version should be shown by default in the output, but still allow for the implementation to switch to (or note or display in some form, such as a tooltip) the original version. This may be documented with a behaviour attribute, for example, of alternate depending on the function library being used. In documenting the processing model for <choice> there are three common pairs of <sic> and <corr>, <abbr> and <expan>, as well as <orig> and <reg> to record:

<model predicate="sic and corr"  behaviour="alternate">  <param name="default">corr</param>  <param name="alternate">sic</param> </model> <model predicate="abbr and expan[1]"  behaviour="alternate">  <param name="default">expan</param>  <param name="alternate">abbr</param> </model> <model predicate="orig and reg"  behaviour="alternate">  <param name="default">reg</param>  <param name="alternate">orig</param> </model>

This example provides explicit behaviour parameters as named <param> children of the <model>. The <param> elements provide parameters to the behaviour implementation documented by the behaviour attribute and their content is always assumed to be an XPath expression. Thus in the example above, we specify expan[1] to choose the first of any available set of <expan> children. The behaviours and their parameters may be documented separately for a customization using a <paramSpec> element. If no <param> is supplied, or it is ., the content of the object documented by this <model> and subject to the behaviour is assumed to be the content of the current node including all child nodes of the element, its attribute nodes, text nodes, etc.

When a different behaviour is desired for other outputs this may be specified with the output attribute. In the following example the processing intention for the <date> element is documented where this has options when containing both regularized and human readable data. This example copes with three situations but could be expanded. It records what to do when:

  1. there is text inside the element, and the output is print;
  2. there is no text inside the element but there is a when attribute, and the output is print; and
  3. there is a when attribute and text inside the element, and the output is web

In the third case, the same alternate function as in the previous example is used, but for print we document that the when attribute should be used when there is no text.

<model output="print" predicate="text()"  behaviour="inline"/> <model output="print"  predicate="@when and not(text())" behaviour="inline">  <param name="content">@when</param> </model> <model predicate="@when" output="web"  behaviour="alternate">  <param name="default">.</param>  <param name="alternate">@when</param> </model>

When documenting the output for elements with attributes, the processing model documentation can access these attributes and their values using standard XPath notation (either abbreviated or unabbreviated syntax). For example, the <ref> element usually has a target attribute providing the location to which a link should be created. The <ref> element may or may not have text content, but sometimes just the target. The processing model documentation below caters for these two situations:

<model predicate="not(text())"  behaviour="link">  <param name="content">@target</param>  <param name="link">@target</param> </model> <model behaviour="link">  <param name="content">.</param>  <param name="link">@target</param> </model>

Here the link function has two <param> elements, the text to be shown (content), and the URL to which it should be linked (link). The first of these <model> elements documents that when there is no text nodes inside the <ref> element it should use the content of the target attribute as if it was textual content. This predicate attribute could be made more specific as not(text()) and @target to assure that there is a target attribute. The second of these uses this target attribute for providing the link, but the element itself (and any text nodes or child elements) of the <ref> element for its content.

The documentation for the suggested model behaviours is provided in the element specification for the <model> element. For example, the inline behaviour has two optional parameters content and label:

<valItem ident="inline">  <paramList>   <paramSpec ident="content"/>   <paramSpec ident="label"/>  </paramList>  <desc versionDate="2015-08-21"   xml:lang="en">creates inline element out    of <term>content</term> if there's something in <gi>outputRendition</gi>,    use that formatting otherwise just show text of selected content.</desc> </valItem>

This documentation inside the <elementSpec> for the <model> element because it is there that we specify the list of attribute values that behaviour allows. Similarly the value for the behaviour attribute alternate has two parameters: alternate and default:

<valItem ident="alternate">  <paramList>   <paramSpec ident="default">    <desc versionDate="2015-08-21"     xml:lang="en">preferred content</desc>   </paramSpec>   <paramSpec ident="alternate">    <desc versionDate="2015-08-21"     xml:lang="en">alternate content</desc>   </paramSpec>  </paramList>  <desc versionDate="2015-08-21"   xml:lang="en">create a specialized display    of alternating elements for displaying the preferred version and an alternative,    both at once or by some method of toggling between the two.</desc> </valItem>

The nature of the documentation of these behaviour parameters intentionally makes no reference to the processing functions which might implement since that is a decision that should be left to the implementor and might vary by choice of processing language and form of output.

2.5. Implementing a TEI Processing Model

It is a principle of the processing model that the transformation stage should be minimally intelligent; that is to say, it should contain as few TEI-specific rules as possible. The ODD should be as explicit as possible, and provide maximum expressivity to the editor without recording details of implementation.

An implementation of a TEI Processing Model can take at least two approaches. One can read the TEI ODD customization in advance and produce a processing pipeline (which itself consults the ODD or not), or a processing implementation could consult the TEI ODD customization for information while processing a document instance. Software which implements this processing model might proceed as follows:

  1. When processing an element in the instance document, any applicable <model> or <modelSequence> for the current output format should be followed whether this is done by consulting the TEI ODD customization directly or rules abstracted into the processing toolchain.
  2. The <model> elements should be processed in order until a match with the predicate is found, or a <model> with no predicate applies. If a <model> has no predicate, it matches any occurrence of the element. If a <model> has no output, it applies to all forms of output. If there is not a useSourceRendition attribute of true on the <model> then any rendition attribute on the source element can be ignored.
  3. All of the <model> elements inside a <modelSequence> should be applied in order if the context matches any optional limiting predicates
  4. The processing should follow the function provided by the behaviour attribute. Where the parameter passed to that function is '.' or no <param> is supplied, then the object of the behaviour attribute is all child nodes of the element including its attributes, element, and text nodes. These nodes should be subject to processing within this context. Thus a block should create a block-type object in the output for this element and all the results from processing its children.
  5. If there are possible <model> elements, but none has a predicate which matches the current element, processing should proceed to its child nodes and see if there are models for them. Likewise, if there is no <model> provided for the current element, proceed to its children and see if there are models for them.
  6. The special behaviour omit should be interpreted to direct processing to stop for that element and to proceed to the next sibling.

While this implementation system is lightly sketched it should be interpreted as broadly similar to the processing model for XSLT.
